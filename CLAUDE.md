# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A **front-end-only clickable prototype** of an in-platform written-submission and text-review feature for **MOSO InPraxis** (Supervision / Veiledning module), built for client concept verification. It is modeled on Canvas SpeedGrader + DocViewer. No backend, no auth, no persistence — all data is mock/in-memory and resets on reload.

The full product spec lives in `src/imports/llm_build_brief.md` (source ticket: Jira MIPU-44). **Read it before making feature changes** — it defines the roles, statuses, user stories, and acceptance criteria the prototype is meant to satisfy.

## Commands

```bash
npm i          # install (or: pnpm i — see pnpm notes below)
npm run dev    # start Vite dev server
npm run build  # production build (vite build)
```

There is **no linter, no test suite, and no typecheck script** defined. There is no git repository initialized here.

## Architecture

**The entire application is one file: `src/app/App.tsx` (~2400 lines).** `src/main.tsx` just mounts `<App />`. When making changes, this is almost always the only file you edit.

Structure within `App.tsx`, top to bottom:
- **Types** — `Role`, `DeliveryStatus`, `AssessmentStatus`, `AnnotationType`, and the `User` / `Submission` / `Annotation` / `Comment` interfaces.
- **Mock data** — `USERS`, `TASK`, `SUBS_INIT`, `ANNS_INIT`, `COMS_INIT` are module-level consts seeded at the top. This is the single source of demo data (there is no separate `mockData.ts` despite the brief calling for one).
- **Atom components** — `Avatar`, `Lozenge`, `Btn`, `ConfirmModal`, `ToastStack`, `AudioPlayer` (mock player, no real audio).
- **Annotation rendering** — `AnnSpan` / `renderAnnotatedPara` / `AnnotatedText`. Annotations are anchored by **substring match** (`text.indexOf(ann.selectedText)`), not by real offsets/ranges — overlapping selections are dropped.
- **Role views** — `TeacherOverview` (sortable table), `TeacherDetail` (annotate + comment + assess + reopen), `FeedbackSidebar` (shared read-only feed), `StudentView` (write/submit/resubmit + read feedback), `SupervisorView` (read-only).
- **`App` (default export)** — owns all state (`submissions`, `annotations`, `comments`, `toasts`, current `role`, view mode) and all mutation handlers. Renders the top nav with the **role switcher** and routes to the active role's view.

**State model:** everything is `useState` in `App`, threaded down via props. There is no router (`react-router` is a dependency but unused), no context, no external store. Navigation is state flags: `role` (`teacher`/`student`/`supervisor`), `viewMode` (`overview`/`detail`), `selectedSubId`, etc.

**Two independent status axes** (do not conflate them):
- `deliveryStatus`: `missing` | `late` | `submitted` — derived from deadline + student action.
- `assessmentStatus`: `not_assessed` | `resubmission_requested` | `approved` | `not_approved` — teacher-set. Only the teacher role can change it. A final comment is required before approve/reject.

**Role gating** is UI-only (the role switcher). In production this must be enforced server-side — the prototype does not.

## Styling — important divergence from the brief

The brief mandates the Atlassian Design System (Atlaskit components + ADF document format). **The actual implementation does not use Atlaskit.** Instead every component is hand-rolled with **inline `style={{}}` objects** that manually replicate the Atlassian color palette (e.g. `#0052CC`, `#172B4D`, `#DFE1E6`) and lozenge/button conventions. Submission bodies are plain strings with `\n\n` paragraph breaks, not ADF JSON. Match this existing inline-style convention when editing `App.tsx` rather than reaching for a component library.

**`src/app/components/ui/*` is unused scaffolding.** It is a full shadcn/Radix component set auto-generated by Figma Make; `App.tsx` imports none of it. Most `package.json` dependencies (Radix, MUI, recharts, embla, etc.) are likewise unused by the actual app. Do not assume a dependency is wired up just because it's installed.

## Build config notes

- `vite.config.ts` aliases `@` → `./src` and includes a custom `figma:asset/` import resolver mapping to `src/assets` (dir may not exist yet). Tailwind and the React plugin are both present; per the config comment, do not remove them even though Tailwind is not actively used by `App.tsx`.
- `pnpm-workspace.yaml` sets `minimumReleaseAge: 10080` (refuses packages published less than 7 days ago) — expect this if adding/updating deps via pnpm.
- Styles are wired through `src/styles/index.css` (imports `fonts.css`, `tailwind.css`, `theme.css`), but the app's visuals come almost entirely from inline styles.

## Editing guidance specific to this repo

- **PDF export** is `window.print()` (the "Download PDF" button) — a mock, per the brief's out-of-scope list. Real PDF generation is not expected.
- Adding a demo student/submission means editing the `USERS` and `SUBS_INIT` consts together (a `Submission.studentId` must reference a `User.id`).
- IDs for runtime-created annotations/comments use `uid()` (`Math.random`); seeded IDs are hand-written (`sub1`, `ann1`, …).
- Keep it a demo: favor visible, clickable behavior over real validation, persistence, or edge-case handling.
